Multimodal Action Recognition using Graph Convolutional Neural Networks

Master Thesis
Graph convolutional neural networks are applied in state-of-the-art models for skeleton-based action recognition. The focus of this work is to research the combination of this type of neural network and data from multiple modalities. Using skeletal data as a base for a graph convolutional network model, multiple approaches for fusion of skeleton, RGB and other sensor modalities are proposed. Experiments are conducted on the two multimodal datasets UTD-MHAD and MMAct using various graph-like representations of modalities and their fusion at different stages of the neural network architecture.

The image shows one of the proposed fusion techniques concerning skeletal data (dark nodes), obtained from OpenPose, Kinect SDK, etc., and signals from accelerometer and gyroscope (red nodes) which are worn by the person performing an action. If both skeleton joints and IMU signals are represented by three-dimensional values (xyz), the skeleton graph can be augmented by simply appending additional nodes representing IMU (or other modalities).

Models are implemented and trained using PyTorch 1.6 with Python version 3.8 and CUDA 10.2. All computations and experiments are conducted on a single Nvidia RTX 2080 GPU with 8GB of video memory. To guarantee a deterministic and reproducible behavior, all training procedures are initialized with a fixed random seed.

Regarding the skeleton-based approach as the base model, experiments to investigate benefits of multimodal approaches show that multiple fusion methods of skeleton and inertial sensor data achieve better classification accuracies than the base model on UTD-MHAD while a slight decrease in performance is noticeable on MMAct. The approaches for fusion of skeleton and RGB, while experiencing high accuracies on several training runs, similarly show a slight decrease in performance over the base model in the majority of cases.

Robbie

During the research internship "Robbie" our team of nine students developed behavior and human-machine interactions for two service robots that are the size of a small human. I was primarily responsible for implementing a robust people following routine for the scenario of carrying groceries or luggage for a human subject. My specific team role was "Quality Management" which involved preparing, supervising and documenting test scenarios according to official tournament rules and ratings as well as introducing and verifying coding guidelines. The project was presented online as part of CV Tag 2020.

People tracking, which is required for following people, is a challenging subject. The task is to identify the same person within an arbitrary number of time-shifted image frames and to prevent a mismatch, that is, recognizing something/someone else as the target. Since this is a visual recognition task, re-identification is further complicated by occlusion (esp. by other people or taking abrupt turns through doors), lighting, video quality and clothes/appearance in combination with viewing a person from different angles. Furthermore, following requires the robot to drive behind the operator at a comfortable distance while avoiding obstacles which inevitably results in it losing its operator.

Using ROS with C++, two packages people-tracking and people-following are created that communicate with each other. The former is responsible for tracking people between frames based on RGB-D images from Kinect, OpenPose detections and OpenCV. A feature vector is computed for each detection and matched with all existing tracks using Hungarian algorithm. Similarity metrics are euclidean distance of last known positions, euclidean distance of predicted positions using Unscented Kalman Filter, histogram comparison using Bhattacharyya distance, distance of face detection embeddings and an online classifier based on this paper (using features of a person reid neural network forwarded into online AdaBoost). The most reliable method is face detection which is not very useful to re-identify a followed person when the robot drives behind its target. people-following subscribes to the robot's base pose and camera pose and receives tracking data to publish commands that adjust the camera's position, turn the robot or drive to a location.

The above picture is produced by the people-tracking package for debugging/visualization purposes.

  2. Person bounding box (blue in tracking-only mode; In following mode: green for followed person, red for everyone else)
  3. Face detection bounding box

Edutainment: Fresco Restauration in VR

In this VR simulation you take the role of a painter who is tasked to restore a fresco ceiling painting inside a virtual reconstruction of Engers castle's Dianasaal. While playing, the game provides explanations and background to both fresco painting and the painting (named "Triumpf der Diana") itself. This project was created in a two-person team over nearly two semesters in Unreal Engine as part of the module Gamedesign/Edutainment. The project was presented as part of CV Tag 2019.

I was responsible to create most of the game's logic, especially the painting system. First, a masked "destruction" texture is applied onto the ceiling painting (see mask in left image below). Painting is accomplished by taking a "brush" texture and projecting it onto a drawable texture (render target) on the ceiling based on the user's input (click location with mouse or joystick direction in VR). The render target (see right image below) itself is then treated as a mask for the destruction texture.

Reconstruction of the painting is divided into two stages. For a fresco, the first step is to apply multiple layers of wet lime plaster that need to dry. However, for this game, only a single layer needs to be applied without waiting for it to dry 😉.
The second stage involves re-coloring the painting in a "color by numbers"-like manner. This time, a render target (mask) is required for each of the fields that are able to be filled with color. To prevent pixel-precise accuracy for completing the painting, a C++ script is used to check whether a mask is "fully colored" by introducing a threshold, e.g., 80% of the pixels need to be filled for the painting to be considered complete.

Realtime Terrain Rendering using Quadtree LOD

Terrain Rendering implemented using C++, OpenGL and tessellation shaders. A Quadtree deals with the LOD of the terrain and is updated whenever the camera moves. Each quadtree cell and its neighbor information is then passed to the tessellation shader where it is appropriately tessellated based on distance to camera.

This way, using the neighbor information, vertex cracks in the environment between two small squares adjacent to one big square are easily avoided. The vertical displacement is then given by sampling a heightmap. While closing cracks is trivial, one big problem using Quadtrees for LOD is vertex popping when four small squares collapse into one big one and vice versa. Research: Logic Puzzles - The Boy or Girl Paradox

The Boy or Girl paradox raises the question about the probability for a random two-child family to have two boys or two girls given some statement about the family or one of the two children. One of the most frequently used statements is about a random two-child family where at least one of the children is a boy and asked for is the probability of them having two boys. When intuitively answering these types of questions, people's opinions about the correct solution and method of achieving said solution differ. When dealing with conditional probabilities, people often apply simple mental models that either lead to a wrong solution or may be correct for a particular problem but don't transfer to other problems.

Statements 1 and 2
  1. Mr. Smith has two children. At least one of them is a boy. What is the probability P(BB) that both children are boys? Answer:
  2. Mr. Jones has two children. The older child is a girl. What is the probability P(GG) that both children are girls? Answer:

The statement "at least one boy" implies that the latter of the four possibilities, girl-girl (see image), is unreachable. This leaves three equiprobable cases, one of which corresponds to the target scenario, i.e., the probability of having two boys. As for statement 2, two of the four cases are unreachable and so the resulting probability is 1/2. However, the first statement is ambiguous and, depending on context and background information, the answer could also be 1/2. The change in outcome is subject to the way in which the information "at least one of them is a boy" is obtained.

Statements 3 and 4
  1. Mr. Smith is the father of two. We meet him walking on the street with a young boy whom he proudly introduces as his son. What is the probability that Mr. Smith's other child is also a boy? Answer:
  2. We meet Mr. Smith (whom we know to be the father of two) in the street with a boy. This time, he is more elaborate in his introduction, presenting the boy as his eldest child. What is the probability that Mr. Smith's other child is also a boy? Answer:

Contrary to what one intuitively believes when comparing statements 3 and 4 to statements 1 and 2, the solution to both statement 3 and statement 4 is identical, that is, P(BB) = 1/2. The relevant information is the observation that we meet the son on a walk with his father. Research: BRDF Importance Sampling

The rendering equation can be approximated using Monte Carlo Integration (MCI) with N samples and a probability density function (PDF) that defines the distribution of incident light directions. Assuming the PDF corresponds to a uniform distribution, the incident light ray has equal probability to come from any direction. This results in a large variance for MCI and would be inefficient for strongly reflective surfaces.
Because convergence to a correct solution is O(n-1/2), halving the approximation error requires four times the amount of samples. Importance Sampling is a technique to pick samples from distributions that reduce variance for MCI and, with that, reduce noise in the rendered image. The figure shows uniform sampling (left) and cosine-weighted sampling (right).

Thesis: Strategien zur Reduktion von Zustandswechseln in modernen OpenGL-Computergrafik-Architekturen in Java

Bachelor Thesis
For my bachelor thesis I developed a mini realtime graphics engine using only OpenGL and Java (JOGL). Since OpenGL is a state machine the goal was to create an efficient architecture and implement techniques to reduce the number of state changes. The architecture is based on the Entity–component–system architectural pattern, similar to Unity Engine.
A naive approach for rendering geometrical objects in OpenGL is traversing a list and changing state and parameters for every object encountered in that list. This turns out to be ineffective, resulting in hotspots at OpenGL functions which require more than 50% of the application's available time, leaving less time for other subsystems of a graphics engine. Multiple techniques are considered and their change performance evaluated: frustum culling, spatial data structures, automated batching of similar objects + multi draws, component sorting, dirty flags to avoid re-computation of unchanged values.

3D Simulation and Manipulation of Signal Representations and their Fourier Transformation in VR

In a two-person team we created a VR-simulation in Unity Engine that lets you modify images using a variety of different brushes and observe their changes influence the magnitude image of its Fourier transformation (or vice versa). I was mainly responsible for rendering and updating the images in realtime, implementing 2D-FFT as a C++ plugin for Unity and create the advisory screen using a HTML5 renderer for Unity.

The image (grayscale only) itself is rendered as a 3D-representation with the pixel intensity being the vertical displacement. For performance reasons, the 3D image is not made up of individual cuboid entities but instead a single mesh that is generated on the fly. The user selects a brush (e.g., 5x5 brush illustrated by green area in image) and applies it by pointing on the image (ray gets projected onto mesh to determine area of input) and then dragging up or down to modify the intensity of the pixel area.
